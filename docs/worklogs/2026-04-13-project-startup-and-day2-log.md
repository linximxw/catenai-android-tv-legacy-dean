## 一、项目启动说明

## 1. 这个项目一开始到底要做什么

这个项目不是去拉一个现成的 4.4 官方分支继续改，也不是把 5.0 工程强行往下兼容。

我们确认后的正确路线是：

- 5.0 私有仓库只是参考快照
- 真正的 4.4 工程要我们自己从零搭
- guide 负责定义接口和流程
- 5.0 快照只负责帮助理解业务语义和字段意义

也就是说，这个项目的开局重点不是“迁移旧代码”，而是“先把 4.4 的地基搭起来”。

## 2. 第一阶段的最小实现是什么

第一阶段最小实现，不是首页，不是完整绑定流程，也不是所有接口都跑通。

第一阶段最小实现是下面这条最短闭环：

1. App 能在 Android 4.4 工程里正常启动
2. 能拿到设备基础信息
3. 能生成设备指纹
4. 能按平台要求统一带请求头发请求
5. 4.4 下 TLS 1.2 不出兼容问题
6. 工程能打出 release APK

先把这条链跑通，后面再接绑定激活、session 校验、配置拉取，成本最低，也最稳。

## 3. 开始时到底先做什么

开头最难的地方，不是写代码本身，而是先把方向定对。

这次启动顺序总结如下：

1. 先确认项目边界
   - 4.4 工程从零搭
   - 5.0 只作参考
   - guide 才是主约束
2. 再搭仓库和分支
   - 建自己的 GitHub 仓库
   - 建 `android-4.4` 基线分支
   - 建 `develop` 和功能开发分支
3. 再准备参考资料
   - 本地 clone 5.0 快照仓库
   - 只当参考，不直接改
4. 再搭 4.4 工程骨架
   - Java/XML
   - `minSdk 19`
   - `targetSdk 28`
5. 再接最小网络底座
   - 设备信息
   - 设备指纹
   - 公共请求头
   - OkHttp
   - TLS 1.2
6. 最后才去接业务接口
   - 先绑定
   - 再 session
   - 再配置拉取

## 4. 一开始要不要先把平台接口全查完

不要一上来把所有平台接口全查一遍。

正确做法是只先核对“当前阶段马上要用到的接口”。

第一阶段只需要重点确认这几类信息：

- 基础域名
- 接口路径
- 请求头要求
- 设备身份字段
- 成功和失败返回语义

这样不会一开始就陷进接口细节泥潭。

## 5. 当前阶段结论

今天结束时，项目已经跨过“开头最难”的那一步。

现在我们已经不是在讨论“怎么开始”，而是在一个已经能编译、能验证的 4.4 基础工程上，继续做 Day3 的绑定流程。

## 四、今天遇到的问题和处理结果

## 1. AndroidX 依赖和编译 SDK 对不上

问题：

- 依赖版本偏新
- `compileSdk` 太低时会直接报依赖元数据不兼容

处理：

- 把 `compileSdk` 提到 34
- 同时保留 `targetSdk 28`

结论：

- 这是 legacy 工程常见做法
- 编译环境可以新，运行目标和行为约束仍然可以按旧项目控制

## 2. Release 打包时 Kotlin 标准库重复

问题：

- 同时带进来了新的 Kotlin 标准库和旧的 `kotlin-stdlib-jdk7/jdk8`
- 打包时类重复冲突

处理：

- 在 `app/build.gradle` 里明确排掉旧的 `kotlin-stdlib-jdk7/jdk8`

结论：

- 问题已经解决

## 3. Release lint 把 legacy 分支当成 Google Play 普通应用

问题：

- `targetSdk 28` 被 lint 当成 fatal 问题拦住 release

处理：

- 只关闭 `ExpiredTargetSdkVersion` 这一条不适用于当前场景的检查

结论：

- 保留 legacy 目标不变
- 同时不去粗暴关闭整套 lint

## 4. 本机没有 Java

问题：

- 本机暂时不能直接跑 Gradle 构建

处理：

- 当前阶段主要用编译服务器做验证

结论：

- 不影响项目推进
- 后续如需要本地独立编译，再补本机 Java 和 Android SDK
